1. Notas sobre Decisiones Pendientes (V1/V2/V3)
Este documento incorpora defaults de MVP para las tres decisiones de infraestructura pendientes de confirmación. Cada sección afectada indica el default usado y la configuración alternativa. Un cambio en cualquiera de estas decisiones requiere únicamente actualizar las secciones indicadas, sin rediseño del stack completo.
V1 — Deployment target (default: Cloud Run):  El documento usa Cloud Run para todos los servicios. Si la fábrica elige Kubernetes, reemplazar los bloques Cloud Run (Sección 3) por los manifiestos Kubernetes incluidos en el Anexo A. El resto del documento no cambia.
V2 — LangGraph checkpointer (default: PostgreSQL / Cloud SQL):  El documento configura PostgreSQL como checkpointer de LangGraph. Si se elige Redis (menor latencia), reemplazar la sección 4.2 con la configuración de Redis incluida en el Anexo B. El DigiKawsayState TypedDict no cambia.
V3 — AG-09 pipeline narrativo (default: texto de viñetas sin imagen):  El documento configura AG-09 sin Gemini Imagen API en el MVP. La variable GEMINI_IMAGEN_ENABLED=false activa el fallback de texto narrativo. Cuando Gemini Imagen esté disponible, cambiar a true en la config de AG-09. Sin cambios en arquitectura.
2. Stack Técnico Completo
DECISIÓN DE STACK:  Todos los servicios son stateless en Cloud Run. El único estado duradero vive en: PostgreSQL (LangGraph checkpoint + Agent Registry + CYCLE_STATE), Weaviate (embeddings + chunks), y GCS (artefactos). Esta separación permite escalar los workers de Cloud Run sin coordinación.
3. Configuración Cloud Run — Servicios
Cada componente del sistema se despliega como un servicio independiente de Cloud Run. Los servicios del enjambre (AG-01 a AG-09) comparten la misma imagen base pero reciben configuración distinta vía variables de entorno, específicamente el agent_id y el pubsub_topic de suscripción.

  4.2  Schema DDL — Tablas Operacionales  
# Schema: digikawsay
# Ejecutar en orden. LangGraph crea su propio schema 'langgraph' automáticamente.
CREATE TABLE projects (
  project_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name              TEXT NOT NULL,
  seed_prompt       TEXT,
  created_at        TIMESTAMPTZ DEFAULT NOW(),
  created_by        TEXT,             -- email del Diseñador (solo en esta tabla)
  status            TEXT DEFAULT 'active' CHECK (status IN ('active','closed','archived'))
);
CREATE TABLE cycles (
  cycle_id          SERIAL,
  project_id        UUID REFERENCES projects(project_id),
  phase             TEXT CHECK (phase IN ('INVESTIGACION','ACCION','PARTICIPACION','CLOSED')),
  started_at        TIMESTAMPTZ DEFAULT NOW(),
  closed_at         TIMESTAMPTZ,
  n_participants_active INTEGER DEFAULT 0,
  n_dialogue_packets INTEGER DEFAULT 0,
  saturation_reached BOOLEAN DEFAULT false,
  saturation_index  FLOAT DEFAULT 0.0,
  consensus_reached BOOLEAN DEFAULT false,
  active_agents     TEXT[],
  PRIMARY KEY (project_id, cycle_id)
);
CREATE TABLE dialogue_states (
  participant_id    TEXT NOT NULL,     -- OAuth sub claim
  project_id        UUID REFERENCES projects(project_id),
  cycle_id          INTEGER,
  checkpoint_id     TEXT,              -- LangGraph checkpoint ID
  turn_count        INTEGER DEFAULT 0,
  emotional_register TEXT CHECK (emotional_register IN ('OPEN','GUARDED','RESISTANT','DISTRESSED')),
  momentum_score    FLOAT DEFAULT 0.5,
  topics_covered    TEXT[],
  topics_pending    TEXT[],
  saturation_flags  TEXT[],
  active_directive_id TEXT,
  safe_harbor_active BOOLEAN DEFAULT false,
  status            TEXT DEFAULT 'active' CHECK (status IN ('active','timed_out','closed','withdrawn','zombie')),
  last_turn_at      TIMESTAMPTZ,
  PRIMARY KEY (participant_id, project_id, cycle_id)
);
CREATE TABLE data_gaps (
  gap_id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id        UUID REFERENCES projects(project_id),
  cycle_id          INTEGER,
  constructo        TEXT NOT NULL,
  agent_id          TEXT NOT NULL,
  cycles_open       INTEGER DEFAULT 1,
  status            TEXT DEFAULT 'OPEN' CHECK (status IN ('OPEN','RESOLVED','DEAD_END','OUT_OF_SCOPE')),
  created_at        TIMESTAMPTZ DEFAULT NOW()
);
CREATE TABLE evidence_trail (
  evidence_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id        UUID REFERENCES projects(project_id),
  cycle_id          INTEGER,
  finding_id        UUID,
  agent_id          TEXT NOT NULL,
  chunk_id          TEXT NOT NULL,    -- ID en Vector DB
  cita_textual      TEXT,             -- max 300 chars, ya anonimizada
  inference_type    TEXT CHECK (inference_type IN ('OBSERVATION','INFERENCE_1','INFERENCE_2','HYPOTHESIS')),
  created_at        TIMESTAMPTZ DEFAULT NOW()
);
CREATE TABLE agent_registry (
  agent_id          TEXT PRIMARY KEY,
  display_name      TEXT NOT NULL,
  capabilities      TEXT[],
  project_tags      TEXT[],
  pubsub_topic      TEXT NOT NULL,
  health_endpoint   TEXT NOT NULL,
  version           TEXT NOT NULL,
  added_by          TEXT,
  added_at          TIMESTAMPTZ DEFAULT NOW(),
  active            BOOLEAN DEFAULT true,
  runtime_config    JSONB             -- temperatura, max_tokens, etc.
);
CREATE TABLE channel_outbound_messages (
  message_id        UUID PRIMARY KEY,
  in_reply_to       UUID,
  participant_id    TEXT NOT NULL,
  project_id        UUID,
  channel           TEXT,
  latency_ms        INTEGER,
  sla_breached      BOOLEAN DEFAULT false,
  created_at        TIMESTAMPTZ DEFAULT NOW()
);
-- Índices críticos para performance
CREATE INDEX idx_dialogue_states_project ON dialogue_states(project_id, cycle_id);
CREATE INDEX idx_dialogue_states_status ON dialogue_states(status) WHERE status = 'active';
CREATE INDEX idx_data_gaps_open ON data_gaps(project_id) WHERE status = 'OPEN';
CREATE INDEX idx_evidence_trail_finding ON evidence_trail(finding_id);
CREATE INDEX idx_agent_registry_caps ON agent_registry USING GIN(capabilities);
CREATE INDEX idx_agent_registry_active ON agent_registry(active) WHERE active = true;
CREATE INDEX idx_outbound_sla ON channel_outbound_messages(sla_breached, created_at);
V2 — Redis alternativo:  Si se elige Redis como checkpointer de LangGraph en lugar de PostgreSQL, las tablas dialogue_states se mantienen en PostgreSQL pero el checkpoint de LangGraph (DigiKawsayState) migra a Redis. Ver Anexo B para la configuración Redis.
5. Vector DB — Weaviate
Weaviate gestiona los embeddings del corpus y sirve las operaciones RAG de todos los agentes. Se despliega como servicio Cloud Run en el MVP. Para producción con alto volumen, migrar a Vertex AI Vector Search o Weaviate Cloud Services.

13. Estado del Documento y Anexos
Anexo A — Nota sobre Kubernetes (si V1 cambia de Cloud Run)
Si se elige Kubernetes en lugar de Cloud Run, los cambios son exclusivamente en los bloques de la Sección 3. Cada service Cloud Run se convierte en un Deployment + Service + HorizontalPodAutoscaler de Kubernetes. El resto del documento (Pub/Sub, PostgreSQL, Weaviate, GCS, IAM, Agent Registry) no cambia. Los manifiestos K8s equivalentes seguirán el mismo patrón de variables de entorno con SecretKeyRef a Kubernetes Secrets o External Secrets Operator apuntando a Google Secret Manager.
Anexo B — Nota sobre Redis como LangGraph Checkpointer (si V2 cambia)
Si se elige Redis (MemoryStore para Redis en GCP), el DigiKawsayState se almacena en Redis en lugar de PostgreSQL. El impacto es: (1) latencia de checkpoint reducida de ~50ms a ~5ms, relevante para conversaciones muy activas; (2) requiere una instancia MemoryStore (tier Standard para HA, mínimo 2GB para MVP); (3) las tablas operacionales (cycles, data_gaps, evidence_trail, agent_registry) permanecen en PostgreSQL sin cambio; (4) en el código de AGENTE-00 y VAL, reemplazar SqliteSaver o PostgresSaver de LangGraph por RedisSaver. La interfaz del checkpointer es la misma; solo cambia la clase de persistencia.

Las siguientes dos tablas se añaden al schema DDL de PostgreSQL de la Sección 4.2. Son parte del mismo schema 'digikawsay' y deben crearse en el mismo script de bootstrap.
-- ═══════════════════════════════════════════════════════
-- TABLAS DE COMPENSACIÓN DISTRIBUIDA (añadir a Sección 4.2)
-- ═══════════════════════════════════════════════════════
-- Outbox Pattern: garantiza atomicidad entre LangGraph checkpoint
-- y Pub/Sub dispatch. AGENTE-00 escribe aquí EN LA MISMA TX que el checkpoint.
CREATE TABLE digikawsay.outbox (
  envelope_id   UUID PRIMARY KEY,
  topic         TEXT NOT NULL,
  payload_json  JSONB NOT NULL,
  status        TEXT DEFAULT 'PENDING'
                CHECK (status IN ('PENDING','DELIVERED','FAILED')),
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  delivered_at  TIMESTAMPTZ,
  retry_count   INTEGER DEFAULT 0,
  last_error    TEXT
);
CREATE INDEX idx_outbox_pending
  ON digikawsay.outbox(created_at)
  WHERE status = 'PENDING';
-- NOTIFY para que el Outbox Worker reaccione inmediatamente
-- en lugar de esperar el siguiente poll de 5 segundos.
CREATE OR REPLACE FUNCTION digikawsay.notify_outbox()
RETURNS TRIGGER AS $$
BEGIN
  PERFORM pg_notify('outbox_pending', NEW.envelope_id::text);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_outbox_notify
  AFTER INSERT ON digikawsay.outbox
  FOR EACH ROW EXECUTE FUNCTION digikawsay.notify_outbox();
-- Idempotency Keys: cada agente especialista registra los
-- envelope_ids que ya procesó para descartar duplicados de Pub/Sub.
CREATE TABLE digikawsay.processed_envelopes (
  agent_id      TEXT NOT NULL,
  envelope_id   UUID NOT NULL,
  processed_at  TIMESTAMPTZ DEFAULT NOW(),
  output_id     UUID,  -- FK lógica a evidence_trail.output_id
  PRIMARY KEY (agent_id, envelope_id)
);
-- Índice para purga periódica (retener 7 días, purgar el resto)
CREATE INDEX idx_processed_envelopes_age
  ON digikawsay.processed_envelopes(processed_at);

AMENDMENT — Sección 3.8: Outbox Worker Service
El Outbox Worker es un servicio Cloud Run adicional (o Cloud Run Job con schedule) que lee la tabla outbox y publica a Pub/Sub. Es el único componente autorizado a publicar Task Envelopes a los topics del enjambre.
# outbox-worker — Cloud Run Service
# Responsabilidad única: leer outbox WHERE status='PENDING'
# y publicar a Pub/Sub. LISTEN/NOTIFY para reacción inmediata.
---
metadata:
  name: digikawsay-outbox-worker
  labels:
    component: outbox-worker
spec:
  template:
    metadata:
      annotations:
        autoscaling.knative.dev/minScale: '1'   # Siempre activo
        autoscaling.knative.dev/maxScale: '3'   # No necesita escalar mucho
    spec:
      timeoutSeconds: 300
      serviceAccountName: digikawsay-ag00-sa   # Misma SA que AGENTE-00
      containers:
        - image: gcr.io/PROJECT_ID/outbox-worker:VERSION
          resources:
            limits: { cpu: '0.5', memory: 256Mi }
            requests: { cpu: '100m', memory: 128Mi }
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef: { name: cloud-sql-url, key: latest }
            - name: OUTBOX_POLL_INTERVAL_SECONDS
              value: '5'
            - name: OUTBOX_MAX_RETRIES
              value: '5'
            - name: OUTBOX_RETRY_BACKOFF_SECONDS
              value: '30'
            - name: OUTBOX_ALERT_PENDING_MINUTES
              value: '5'
            - name: GCP_PROJECT_ID
              value: 'PROJECT_ID'
# Comportamiento del worker:
# 1. LISTEN outbox_pending (PostgreSQL NOTIFY)
# 2. Al recibir notificación O cada 5s:
#    SELECT * FROM outbox WHERE status='PENDING' ORDER BY created_at LIMIT 50
# 3. Para cada envelope: Pub/Sub publish(topic, payload_json)
# 4. Si publish OK: UPDATE outbox SET status='DELIVERED', delivered_at=NOW()
# 5. Si publish FAIL: UPDATE outbox SET retry_count=retry_count+1, last_error=...
#    Si retry_count >= 5: SET status='FAILED', emit ALERT
Impacto en agente00-service (modifica Sección 3.5)
AGENTE-00 ya NO publica directamente a los topics del enjambre. En su lugar, escribe en outbox dentro de la misma transacción del checkpoint. El siguiente env var debe añadirse a la configuración de agente00-service:
# Añadir a agente00-service env vars (Sección 3.5):
- name: DISPATCH_MODE
  value: 'OUTBOX'   # 'DIRECT' desactiva el Outbox Pattern (solo para testing local)
- name: OUTBOX_TABLE
  value: 'digikawsay.outbox'
IMPORTANTE:  El Outbox Worker debe deployarse ANTES de AGENTE-00 en cualquier despliegue nuevo. Si AGENTE-00 arranca con DISPATCH_MODE=OUTBOX y el Outbox Worker no está activo, los Task Envelopes se acumularán en outbox sin ser entregados.
